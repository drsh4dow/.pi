# Game Design

First move, before any design or critique: establish **who the game is for** and **what its hook is**. If the user hasn't said, ask — one question. Audience sets the difficulty target; the hook is covered in [PITCH.md](PITCH.md). Every judgement below is relative to that audience.

For hooks, Steam pages, pitches, capsules, or trailers, read [PITCH.md](PITCH.md) before answering.

## Fun is learning (Koster, *A Theory of Fun*)

Fun is the brain's reward for grokking a pattern. A mechanic is fun only while the player is still learning it, and dies the moment the pattern is fully absorbed. So interrogate every mechanic with: **what is the player learning, and how long does that learning last?** Depth = time-to-mastery. If mastery takes one attempt, deepen the mechanic or cut it. Noise (pure randomness with nothing to learn) and rote repetition (nothing left to learn) both read as boring for the same reason: no pattern in motion.

## The flow channel (Csikszentmihalyi, *Flow*)

Hold challenge just above the player's current skill. Always winning is boredom; always losing is anxiety; the channel between them is where play lives, and it must rise as skill rises. The channel's width is audience-relative: a *Getting Over It* or Souls audience wants the ceiling of the channel; a cozy audience wants the floor. Flow also requires **clear goals** and **immediate feedback** — when a game feels off, check those two before touching difficulty numbers, since a fair challenge with murky goals still feels unfair.

## Core loop

Name the moment-to-moment loop as verbs (e.g. aim → shoot → loot), then the session loop and meta loop it feeds. Every mechanic must feed a loop; an orphan mechanic is a cut candidate, however clever. In critique, drawing the loops is the fastest way to expose why a game drags.

## Progression

Introduce one pattern at a time, then escalate by *combining* already-learned patterns rather than only inflating numbers — combinations are new patterns to grok, stat inflation is not. The first minutes carry the whole game: teach through play, in a safe space, with the hook reached fast. Late game, mastered patterns need remixing (modifiers, enemy pairings, constraints) or the flow channel flattens into grind.

## Game feel and juice (Swink, *Game Feel*)

Feel = real-time control + perceptible consequence. The core verb must feel good in an empty room before any content is layered on. Juice — screenshake, hit-stop, tweened motion, sound layers, particles — is disproportionate feedback per input; it is cheap and players read it as quality. When a prototype "isn't fun yet," juice the core verb before redesigning mechanics: dead feedback masquerades as dead design.

## Prototypes: find the fun

Prototype the core loop ugly and fast; production value hides the answer. Toy test: strip goals — is the core verb still amusing to fiddle with? Playtest with strangers, watch silently, and treat what they *do* as the data, since what they *say* is politeness. The prototype has answered its question when you can state what the player learns, which loop it proved, and whether the fun survived a stranger.

## Critique bar

A design critique is complete only when every mechanic is traced to (a) the pattern the player learns from it, (b) the loop it feeds, and (c) its position in the flow channel for the stated audience — plus a verdict on the hook per PITCH.md. Mechanics that fail (a) or (b) get named as cuts, with the reasoning.
